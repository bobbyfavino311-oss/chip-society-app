CURRENT PROBLEM

The Fortune Cookie system is not respecting tier value.

Low-tier cookies are still giving rewards that are too valuable.

The reward table appears to be mixed, shared, or incorrectly weighted across tiers.

This breaks the entire reward economy.

Problems to fix:

* Common cookies are giving rewards that should only belong to higher tiers.
* Low-tier cookies can sometimes feel better than rare cookies.
* Rewards are not properly separated by tier.
* The system does not feel fair, predictable, or trustworthy.
* The prize values do not match rarity.
* The reward logic needs to be rebuilt, not lightly adjusted.

CORE RULE

Each cookie tier must have its own separate reward pool.

Do not use one shared reward table.

Do not allow low-tier cookies to pull from high-tier rewards.

Do not allow random prize values outside the allowed range for that tier.

Higher tier must always mean higher reward value.

REWARD LOGIC REQUIREMENTS

The cookie opening logic must work in this exact order:

1. Determine cookie tier.
2. Load only that tier’s reward pool.
3. Select reward type from that tier’s allowed rewards.
4. Select reward amount inside that tier’s min/max range.
5. Display reward.
6. Grant reward to player account.
7. Save reward server-side.
8. Log the reward for testing.

Never determine reward value before determining tier.

Never use a global reward table.

Never allow fallback rewards from another tier.

Never use the same reward array for multiple tiers.

⸻

SERVER-SIDE VALIDATION

All Fortune Cookie rewards must be validated server-side.

The client should not be trusted to determine prize value.

Server must check:

* Cookie tier
* Allowed chip range
* Allowed ticket range
* Allowed XP range
* Allowed cosmetic reward pool
* Player inventory
* Reward history

If a reward is outside the allowed tier range, reject it.
